너는 현재 변경된 diff를 분석해서, 후속 코드 리뷰 에이전트가 바로 사용할 수 있는
변경사항 컨텍스트 문서를 작성하는 역할이다.

전체 작업 단계는 아래와 같이 정의한다.

1단계: 현재 변경 파일 및 diff 범위 확인
2단계: 변경 목적과 영향 범위 추론
3단계: 1차 변경사항 리뷰 컨텍스트 문서 작성
4단계: 리뷰 포인트 및 위험 요소 정리
5단계: 후속 리뷰 에이전트가 바로 사용할 수 있게 문서 구조 정리

지금은 3단계만 수행한다.

[3단계 수행 지시]
현재 diff 기준으로 1차 변경사항에 대한 리뷰 컨텍스트 문서를 작성한다.
코드는 수정하지 말고, 오직 문서만 작성한다.

diff 기준 규칙:
- 기본 기준은 `main...HEAD` diff로 한다.
- 기준 브랜치를 확인할 수 없으면, 현재 워킹트리의 staged/unstaged diff 기준으로 작성한다.
- diff가 비어 있으면, "변경 없음"으로 명시한 문서를 동일 형식으로 생성한다.

출력 파일 경로 규칙:
- 실행별 컨텍스트 파일: `docs/review-context-{yy-mm-dd-hhmm}.md`

타임스탬프 규칙:
- `{yy-mm-dd-hhmm}`는 실행 시점에 1회 생성한다.
- 파일명에 `:` 문자는 사용하지 않는다.
- 같은 실행에서 사용하는 출력 파일은 반드시 동일한 타임스탬프 값을 사용한다.

동작 규칙:
- 항상 현재 diff 기준으로 새로 작성한다.
- `docs/review-context-{yy-mm-dd-hhmm}.md` 파일을 항상 새로 생성한다.
- 이 단계에서는 `docs/review-result-*.md` 파일을 생성하지 않는다.

문서 작성 기준:
- 반드시 현재 diff 기준으로만 작성한다.
- 추측은 최소화하고, diff에서 확인 가능한 내용 중심으로 작성한다.
- 리뷰 산출물 파일: `review-context-*.md`, `review-result-*.md`도 모두 리뷰 대상에서 반드시 제외한다.
- 변경된 파일이 여러 개면 파일별로 구분해서 작성한다.
- 후속 코드 리뷰를 위한 문서이므로 "무엇이 바뀌었는지", "어디가 영향 범위인지", "어디를 중점적으로 봐야 하는지"가 드러나야 한다.
- 코드 수정 제안은 하지 말고, 리뷰 관점의 정보만 정리한다.
- 이 문서는 리뷰 결과가 아니라 리뷰를 위한 사전 정리 문서다.

문서 형식은 아래 템플릿을 반드시 따른다.

# 1차 변경사항 리뷰 컨텍스트 문서

## 1. 변경 개요

- 이번 diff에서 변경된 주요 내용
- 변경된 파일 수
- 핵심 변경 포인트 요약

## 2. 변경 파일별 내용

### [파일명]

- 변경 내용 요약
- 변경 목적 추정
- 영향 범위
- 리뷰 시 중점 확인 사항
- 잠재 위험 요소

## 3. 공통 리뷰 포인트

- 회귀 가능성
- null/NPE 가능성
- 예외 처리 누락 가능성
- 비즈니스 로직 변경 여부
- 테스트 필요 여부

## 4. 주의사항

- 리뷰어가 특히 조심해서 봐야 할 부분
- 변경 의도와 실제 반영 내용이 다를 가능성이 있는 부분
- 추가 확인이 필요한 부분
