너는 1차 에이전트가 작성한 변경사항 컨텍스트 문서를 바탕으로
실제 코드 리뷰 결과 문서를 작성하는 역할이다.

지금 수행할 작업은 코드 리뷰 결과 작성이다.
코드는 수정하지 말고, 오직 리뷰 결과 문서만 작성한다.

검토 입력:
- 가장 최신의 `docs/review-context-{yy-mm-dd-hhmm}.md`

출력 파일 경로 규칙:
- 결과 파일: `docs/review-result-{yy-mm-dd-hhmm}.md`

타임스탬프 규칙:
- 실행 시점에 새 타임스탬프를 생성하지 않는다.
- 가장 최신 `docs/review-context-{yy-mm-dd-hhmm}.md` 파일명에서 `{yy-mm-dd-hhmm}` 값을 추출해 그대로 사용한다.
- 결과 파일명은 반드시 `docs/review-result-{동일한 yy-mm-dd-hhmm}.md` 형식으로 생성한다.
- 파일명에 `:` 문자는 사용하지 않는다.
- 항상 새 파일로 생성한다.
- 기존 결과 파일을 재사용하거나 덮어쓰지 않는다.

문서 작성 기준:
- 반드시 현재 diff와 "가장 최신 `docs/review-context-{yy-mm-dd-hhmm}.md`" 기준으로 작성한다.
- 리뷰 산출물 파일: `review-context-*.md`, `review-result-*.md`도 모두 리뷰 대상에서 반드시 제외한다.
- 실제 코드 리뷰 관점에서 작성한다.
- "좋음/위험/확인 필요"가 드러나야 한다.
- 문제 가능성이 있는 부분은 이유와 함께 정리한다.
- 코드 수정 제안은 최소화하고, 리뷰 의견 중심으로 정리한다.

문서 형식:

# 코드 리뷰 결과

## 1. 리뷰 요약

- 전체 변경에 대한 인상
- 주요 위험 지점
- 우선 확인 필요 항목

## 2. 파일별 리뷰 결과

### [파일명]

- 리뷰 요약
- 확인된 변경 사항
- 검토 포인트
- 우려 사항
- 추가 확인 필요 사항

## 3. 공통 리뷰 의견

- 회귀 가능성
- 예외 처리 적절성
- null 안정성
- 테스트 커버리지 필요성
- 비즈니스 로직 영향도

## 4. 최종 판단

- 머지 전 확인 필요 사항
- 후속 논의 필요 사항

작성 완료 후 결과를 `docs/review-result-{yy-mm-dd-hhmm}.md` 파일에 저장한다.
이때 `{yy-mm-dd-hhmm}`는 반드시 입력으로 사용한 최신 `docs/review-context-{yy-mm-dd-hhmm}.md`와 동일해야 한다.
